Software Reliability
Growth Modeling

A quantitative engine for predicting software stability, defect removal efficiency, and release readiness through non-homogeneous Poisson processes.

01 The Foundation

Software Reliability Growth Models (SRGMs) are mathematical expressions that model the fault removal process during testing and operational phases. The core assumption is that as software is tested and faults are removed, the failure frequency (intensity) decreases over time.

Core Equation (NHPP)

\[ P(N(t) = n) = \frac{[\mu(t)]^n}{n!} e^{-\mu(t)} \]

Where \( N(t) \) is the number of failures observed at time \( t \), and \( \mu(t) \) is the mean value function representing the expected number of failures.

02 Analytical Models

Goel-Okumoto (GO) Model

The GO model assumes that failure intensity decreases exponentially. It is effective for systems where the reliability growth is consistent and fault removal is immediate and perfect.

Mean Value Function

\[ \mu(t) = a(1 - e^{-bt}) \]

Failure Intensity

\[ \lambda(t) = ab \cdot e^{-bt} \]
a: Total expected number of faults
b: Fault detection rate (per hour)

Musa-Okumoto (MO) Model

Also known as the Logarithmic Poisson Execution Time model. It suggests that earlier faults contribute more to the failure rate than later ones, providing a more realistic "pessimistic" decay curve.

Mean Value Function

\[ \mu(t) = \frac{1}{\theta} \ln(1 + \lambda_0 \theta t) \]

Failure Intensity

\[ \lambda(t) = \frac{\lambda_0}{1 + \lambda_0 \theta t} \]

θ: Failure intensity reduction parameter

03 Model Selection Strategy

The Akaike Information Criterion (AIC)

Instead of just checking raw accuracy, we use AIC to balance fitting quality against model complexity. This prevents "overfitting" where a model looks perfect on old data but fails to predict the future.

\[ AIC = 2k - 2\ln(\hat{L}) \]
Decision Rule

A lower AIC indicates a better model. If GO has an AIC of 150 and MO has 142, MO is mathematically superior for the current dataset.
